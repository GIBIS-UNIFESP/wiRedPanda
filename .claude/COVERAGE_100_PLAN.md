**Recurring exclusion patterns to recognize on sight** (don't re-derive from scratch each time):
1. A function's closing `}` shown as uncovered when the function itself clearly ran → compiler-generated cleanup for a local non-trivial-destructor object's exception-unwind path that's never taken. Exclude the closing brace line.
2. A `qCDebug`/`qCWarning` call split across multiple physical lines shows the first line covered but continuation lines `#####` even when the log call is definitely reached → the streamed arguments only evaluate when the category is enabled (default verbosity is 0 in tests), and gcov only credits the *first* line for the always-evaluated enabled-check. Fix: reformat onto one physical line (matches this codebase's existing single-line convention anyway) rather than fiddling with `Comment::setVerbosity()` in tests.
3. `= default` special member functions (esp. virtual destructors) may show as uncovered even when heavily exercised, since the trivial body gets inlined at each call site with no attributable standalone execution. Exclude with a one-line justification rather than chasing it.
4. Before writing a test for an `if (someString.isEmpty())` fallback fed by a helper function, check every return path of that helper first — if it has no path that returns an empty string (common for "resolve to a name or a sensible default" helpers), the fallback is dead and no circuit will ever trigger it.
5. A variable name built as `prefix + counter + ...` can never exactly equal a bare reserved/keyword string — don't spend time hunting a label that would trigger an `isReserved(fullVarName)`-style check if the counter is unconditionally part of that name.
6. When constructing a "run out of a shared resource" test (pins, IDs, etc.) with a MIX of element types, scene iteration order (`QGraphicsScene::items()`, roughly most-recently-added-first) determines which element's own overflow check actually fires — order assumptions are easy to get backwards. The robust fix: make every element competing for the resource the *same type*, so whichever one is starved when the resource runs out throws from the identical source line regardless of order.
7. A `GraphicElement`'s label is *never* genuinely empty once it round-trips through save+load as an IC boundary port — `ICLoader::buildPortLabels()` falls back to the class name (e.g. `"InputSwitch"`) whenever the label is blank, even an explicit `setLabel("")`. Don't assume "leave it unlabeled" reaches an `isEmpty()`/`"_unnamed"` fallback; verify with a `qWarning()` probe first (see the IC dedup test in `TestSystemVerilogCodeGenUnit`) before writing the test or excluding the line.
8. A multi-line statement built from chained calls (`.arg().arg()...`, not just streamed `qCDebug`/`qCWarning`) can show its *first* line as 0-hits while the continuation lines show real counts — the opposite direction from pattern 2, but the same root cause (gcov attributing the statement's "entry" oddly across lines). Fix: reformat onto one physical line; don't assume the surrounding loop is unreached just because the first line reads `#####`.
9. To build a *real* multi-level IC fixture for a unit test (not an integration `.panda` file), construct the sub-circuit directly (`WorkSpace`/`CircuitBuilder`), call `workspace.save(path)`, then `IC::loadFile(path, dir)` — this round-trips through the actual production save/load path (real port-label derivation, real boundary-Node proxying) without needing to hand-craft a `.panda` blob. This is the reusable "minimal IC" helper technique deferred during `ArduinoCodeGen.cpp` and used to close out `SystemVerilogCodeGen.cpp`'s IC-reserved-word/dedup gap.
10. Before assuming an out-of-topological-order element reference is dead/impossible, check whether the surrounding code has an explicit recursive fallback for it (e.g. `otherPortNameImpl()` recursing into `generateLogicExpressionImpl()` when an upstream gate hasn't been assigned yet) — if so, it's live defensive-but-real code for when `Scene`/caller-supplied element order isn't strictly topological, and deserves a real test (build the elements vector directly, order it deliberately), not an exclusion.
11. A `Q_ENUM`/`Q_GADGET` enum's moc-generated reflection registration can have *several* distinct compiled entry points attributed back to the same `Q_ENUM(Name)` source line — one `QMetaEnum` API alone (e.g. just `valueToKey()`) may leave others (`keyToValue()`, `key(int)`, and critically a `qDebug() << value;` stream call) uncovered. Exercise all four before concluding any remainder is unreachable.
12. A **Meyer's singleton** (`static T instance; return instance;` inside a function, e.g. `ThemeManager::instance()`) is constructed exactly once, at whatever moment the whole test binary first touches it — usually very early, well before any test gets a chance to control the global/settings state the constructor reads. Constructor branches keyed on that first-read state (e.g. "was a theme already persisted?") are architecturally locked out from later tests, not dead code — document why, don't chase them, and don't add a test-only reset hook (that's a behavior change, not a test seam).
13. Before assuming a "received/progress exceeds a configured buffer cap" branch is reachable, check empirically with a real local `QTcpServer`/similar test first: `QIODevice`-based backpressure (e.g. `QNetworkReply::setReadBufferSize()`) can make Qt itself refuse to admit more than the cap when nothing drains the device, so the counter may provably never exceed the configured limit — don't assume the surrounding code's own comment about the intended failure mode is achievable from a synchronous test.
14. When a newly-added constructor/friend/member touches a header that's included very widely (e.g. `Application.h`, `Enums.h`), expect the next `cmake --build` to recompile a large fraction of the codebase — this is normal, not a sign something is wrong; let it run in the foreground or background and don't restart it.
15. A shared `m_hasOutputDevice`-style hardware-detection flag (real audio/serial/etc. device present or not) makes coverage of its gated branches depend on whether the CI/sandbox machine actually has that hardware — this sandbox has a real (virtual) ALSA device, so it's always `true` here and could be `false` on a headless CI runner. Add a small friended test-only setter (mirrors `UpdateChecker::setApiUrlForTesting()`) to force the flag deterministically rather than relying on whatever the current machine happens to have.
16. Before assuming a version-gated old-format load branch (`!hasXxx(version)`) needs a real backward-compatibility fixture, check whether the feature it gates could even have existed in a file that old — if the element type itself was introduced at the same version the gate checks (e.g. `hasAudio()` gates both the version *and* the element's own introduction), no genuine `.panda` file predating it can ever contain one, and the branch is only reachable via a hand-built synthetic stream (or is a legitimate exclusion if building one isn't worth the effort). When it *is* worth building: mirror `GraphicElementSerializer::loadOldFormat()`'s exact field order (pos, rotation, label, 4×portsize, trigger, [unusedPriority], inputPorts, outputPorts, [appearanceNames]) and write `0` for port/appearance *counts* to skip their loops entirely rather than hand-crafting valid per-port entries.
17. A hand-built old-format load test using a `":/..."` resource-path string as the "bare path" payload silently no-ops: `ExternalFilePath::forReading()` always returns `std::nullopt` for a resource path under `PortableFile` purpose (by design — a stale saved resource identifier must not clobber the current compiled-in default), so the test passes without ever reaching the `setAudio()`/`setColor()`/etc. call it's meant to cover if the element's default happens to coincidentally match. Use a real absolute (e.g. `QTemporaryFile`) path instead, and assert the loaded value actually changed to it.
18. `resetSimState()` bypasses `setOutputValue()`'s change-detection entirely (it mutates `ElementSimState`'s output vector directly). `Simulation::update()`'s idle-tick-skip optimization only wakes on a clock/input-element's own `outputChanged()` flag, so calling `resetSimState()` directly then a bare `sim->update()` with nothing else touched gets silently swallowed — not a bug, just not a scenario any real caller (`WaveformSimulator`, which always immediately drives real input changes right after its reset loop) produces. Tests must do the same: drive a real input change immediately after calling `resetSimState()` directly.
19. `QDrag::exec()` (native drag-and-drop) is not automatically unsafe to call in a headless-ish test environment — confirmed empirically here (a real X11 `DISPLAY` is set, not `offscreen`) that it returns immediately rather than blocking, since there's no real held mouse button to sustain a drag session. Don't assume it hangs; verify first with a `timeout`-wrapped isolated run before deciding to exclude a `startDrag()`-reaching code path instead of testing it directly.
20. Before writing a test for a `QGraphicsItem`'s protected `mousePressEvent()`/`mouseMoveEvent()`/etc., check whether it's actually a `QWidget` (like `ElementLabel`, a `QFrame`) rather than a scene item — for widgets, `QCoreApplication::sendEvent(&widget, &qMouseEvent)` reaches the protected handler directly (no scene/view needed), but a **no-button `MouseMove`** sent this way may still never arrive: `QApplication`'s own event delivery (not just the widget) filters out untracked, no-button moves before dispatch. Confirm with a temporary debug print if a "no button held" branch stays stubbornly uncovered despite the send — it may be a genuine (if surprising) unreachable branch, not a mistake in the test.
21. Unlike a `QWidget`, a real `QGraphicsItem`/`QGraphicsObject`'s `sceneEvent()` is **not** reachable via `QCoreApplication::sendEvent(item, &event)` — that only calls `QObject::event()`, which for a graphics item does not forward to `sceneEvent()`. A `QGraphicsSceneMousePress` sent straight to a bare `Scene` (no view) also silently goes nowhere: `QGraphicsScene::event()`'s own mouse-press handling needs a real `QGraphicsView`/viewport in the loop (confirmed by a temporary debug probe showing zero dispatch either way). The one proven-reliable route for a real press/release is `QTest::mousePress(view->viewport(), button, modifiers, pos)` through an actual `MainWindow`+`GraphicsView` (see `testInputButtonMousePressReleaseIsMomentary`). If even that is unreliable for a specific modifier combination (see pattern 22), the fallback is a small `friend class TestX;` seam calling the protected `sceneEvent()` directly with a hand-built event — same trade-off as the existing `GraphicElementSerializer`/`IC`/`ICPreviewPopup` friend seams.
22. This sandbox's offscreen/X11 setup does not reliably reproduce a *specific* requested `Qt::KeyboardModifier` on a `QTest::mousePress(..., modifier, ...)`-synthesized scene event — empirically, requesting `Qt::ShiftModifier` arrived at `sceneEvent()` as `Qt::ControlModifier` instead (confirmed via temporary debug probes at the widget, scene, and item level), while `Qt::ControlModifier` itself sometimes didn't reach the item at all. Don't spend time debugging *why*; for a guard keyed on a specific modifier, prefer the direct-`sceneEvent()`-call friend seam (pattern 21) over chasing real synthetic input.
23. `Application::renderingEnabled` is `false` throughout the entire test suite (tests never paint), so `Connection::updatePath()` — and any similarly rendering-gated code — is a genuine, unconditional no-op in tests. A geometry-based assertion after moving a wire endpoint (`boundingRect()`/`sceneBoundingRect()` containing an expected point) will silently fail even when the underlying logic is correct, because the path never actually rebuilds. Use a behavioral assertion instead (which branch/state was reached — e.g. `hasEditedConnection()`/`editedConnection()->endPort()` — not the resulting paint geometry).
24. A Ctrl/Shift-modifier `QGraphicsSceneMouseEvent` synthesized via `QTest::mousePress`/`mouseClick` doesn't reliably carry the modifier through this sandbox's event pipeline — confirmed again (beyond the earlier `sceneEvent()`-specific pattern 22 finding) for a plain `Scene::mousePressEvent()` reached the normal way, through a real `GraphicsView`'s viewport. When a guard is keyed on a specific modifier and a direct `sceneEvent()` friend seam isn't applicable (the handler is a `QGraphicsScene` override, not a `QGraphicsItem` one), add a narrow `friend class TestX;` grant so the test can call the protected event handler directly with a hand-built event whose `setModifiers()` is set programmatically — same trade-off as the existing `sceneEvent()` friend seams.
25. `SceneInteraction::mouseDoubleClick()` (and potentially other double-click handlers) look up "what's under the cursor" via their own internally-tracked last-mouse-position field (updated only by `mousePress`/`mouseMove`), not the double-click event's own position. `QTest::mouseDClick()` alone doesn't reliably prime that field first — send a real `QTest::mouseClick()` at the same point immediately before the `mouseDClick()` call.
26. A wire-detach operation (`ConnectionManager::detach()`) deletes the old completed `Connection` and immediately starts a *new* in-progress one from the same output port — a raw `TestUtils::countConnections()` before/after comparison can't tell these apart (both count as 1 real `Connection` item on the scene), and reads as "detach did nothing" even when it worked correctly. Assert on the *specific* objects instead: the original input port's `connections()` is empty, and the new `editedConnection()->startPort()` matches the original wire's source.
28. `QMenu::exec()` shows the menu as a `Qt::Popup`, not a `QWidget::isModal()` dialog — `TestUtils::AutoDismisser::closeAnyModal()` (which only matches `w->isModal() || qobject_cast<QMessageBox*>(w)`) never detects it, so a test driving code that calls `menu.exec()` hangs indefinitely under that dismisser. Use a menu-specific `AutoDismisser` handler instead: `TestUtils::AutoDismisser([](QWidget *w){ if (auto *menu = qobject_cast<QMenu*>(w)) { menu->close(); return true; } return false; })`. If a background test run produces no output and doesn't return within a small multiple of its normal runtime, suspect this before waiting longer — kill it and check for an `exec()`-based popup/dialog the dismisser doesn't match.
29. `QGraphicsScene`'s base-class event handlers don't reliably `ignore()` an event just because there's nothing for them to do — confirmed empirically (not assumed) that both `QGraphicsScene::dragEnterEvent()`'s and `QGraphicsScene::helpEvent()`'s default/fallback paths left the event `isAccepted() == true` even with no droppable item or tooltip present. Don't assume "unhandled branch → event ends up unaccepted"; find the *actual* differentiating side effect instead (e.g. `dragEnterEvent()`'s handled branch narrows `dropAction()` to a single value via `acceptProposedAction()`, while the base-class fallback leaves it at its unnarrowed multi-flag default — confirmed by a temporary debug print before writing the real assertion). `QGraphicsScene::dragMoveEvent()`'s fallback, by contrast, does leave the event unaccepted — verify per-event-type, don't generalize from one.
31. When a test needs a stack-allocated `QWidget`-derived object reparented to a *second* stack-allocated `QWidget` (e.g. testing `setParentWindow()`-style reparenting), declare **both** possible parents before the child object. C++ destructs locals in reverse declaration order; once reparented, the *new* parent's destructor becomes responsible for deleting the child, so if that parent is declared (and thus destructed) *after* the child, its destructor tries to delete an already-destructed stack object — a real "double free or corruption" abort, not a hang (a piped/redirected run showed no output at all until the abort, since stdout was fully buffered rather than line-buffered; use `stdbuf -oL` when live-tailing a suspected-hanging GUI test to actually see progress).
32. If a `cmake --build` gets interrupted mid-compile (e.g. an accidentally-stacked duplicate invocation gets killed), check for **zero-byte `.gcno` files** afterward (`find build-coverage -name '*.gcno' -size 0`) before trusting the next `gcovr` run — `ccache` can serve a corrupted/incomplete cached `.gcno` for the file that was mid-compile at kill time even though its `.o` compiles and links fine, silently breaking `gcovr` for that one file (and, if the cache is near its size cap, sometimes more than one, discovered one at a time as gcovr processes files). Fix: delete the affected `.o`/`.gcno`/`.gcda`, rebuild once with `CCACHE_DISABLE=1` to confirm, then (if the cache looks generally unhealthy — check `ccache -s`) `ccache -C` and do one full `rm -f **/*.gcno **/*.gcda **/*.o` + rebuild to guarantee every file's instrumentation data is self-consistent before trusting the coverage numbers. Root cause of the corruption in the first place: never re-run a build command before confirming the previous one finished (see standing rule against stacking background builds) — this cost real time to track down.
33. `QDir::entryList(nameFilters, QDir::Files)` (no `QDir::Hidden` flag) excludes dot-prefixed file names by construction — confirmed empirically with a standalone probe against this exact Qt build, not assumed from docs. A defensive "skip files starting with '.'" loop written against such a list is unreachable dead code, not a bug — mark it with `LCOV_EXCL_LINE` rather than fabricating a scenario that can't occur through the real API.
34. Before assuming a `findChildren<T*>("someObjectName")` filter (or an `objectName()`-based regex match) is dead code just because nothing in the immediately-read function sets that object name, grep the *entire* class/file — `setObjectName()` calls are often made by a sibling function (e.g. one code path tags `"label_ic"` while a different, seemingly-unrelated code path is the one that reads it back). If truly nothing sets it anywhere (checked via `git log -S` archaeology if the intent is unclear), this is the one case where ground rule #2 ("fix the bug, don't bend the test") applies even to a search/query feature: adding the missing `setObjectName()` call(s) is a minimal, safe root-cause fix — restoring previously-intended, currently-silently-broken behavior — not a redesign.
35. A class with a **virtual destructor** compiles to up to three distinct ABI symbols sharing the same source line(s) — the base-object destructor (`D2`, invoked by plain scope-exit/stack destruction and by a derived class's destructor), the complete-object destructor (`D1`), and the deleting destructor (`D0`, invoked by `delete` through a pointer — it calls `D1`/`D2` then `operator delete`). `gcov`/`gcovr` aggregate per source line across all three; if every test (and every production caller) only ever stack-allocates the class (common for a modal `QDialog` run via `exec()`), `D0` never executes and the shared destructor line(s) read as **uncovered** despite the stack-destruction variant running dozens of times (confirmed by reading the raw per-symbol `gcov -b` output, which showed `D2` at 16/16 calls and `D0` at 0/0 for the exact same source line). Fix: add one test that heap-allocates (`new`) and `delete`s an instance — cheap, and a legitimate (if currently unused) usage pattern for any class with a public destructor, not a fabricated scenario. Recognize this pattern whenever an otherwise-fully-exercised class's plain `{ }` destructor lines show as missing.
36. To deterministically drive or capture code that reads real `stdin`/writes real `stdout` with no dependency-injection seam (e.g. `QTextStream cin(stdin)` / `QTextStream(stdout)`), redirect the actual file descriptor with `dup`/`dup2` (save the original fd via `dup(0)`/`dup(1)` first, `dup2` it back in the destructor) — **not** `freopen(path, mode, stdin)` followed by `freopen("/dev/null", ...)` to "restore" it. The latter permanently severs the stream from wherever it really pointed; for stdout specifically this silently swallows `QTestLib`'s own PASS/FAIL output for the rest of the test run (confirmed by a real failure: the test binary appeared to hang/crash with no further output, when it had actually kept running fine, just with its output redirected into the capture file). A `QFile`-backed capture file read back with an `fflush()` first is the actual data channel; the RAII guard's only job is fd bookkeeping.
37. Some `.qrc`/`qt_add_translations()` resources are scoped to the production executable target only, by design, and are never embedded in the test binary (`test_wiredpanda`) — confirmed via `CMakeLists.txt` (the `qt_add_translations(wiredpanda ...)` call names only the app target) and the resource system itself (`QDir(":/i18n").exists()` is still true in the test binary regardless, since a *different*, always-shared resource — `wiredpanda_resources`' `/i18n/ExerciseTour` — registers a subdirectory there, but no actual `wpanda_*.qm` files ever appear). Lines that can only run once such a resource is present (loading a real translation, listing real `.qm` files, the "no `:/i18n` at all" fallback branch that also can't trigger since the shared resource guarantees the directory exists) are permanently unreachable in this test binary without a build-system change — asked the user rather than silently deciding to duplicate the translation-compile pipeline onto the test target; confirmed answer was to document and move on. Mirrors the existing `QSKIP`-based acknowledgment already in `TestLanguageManager.cpp` for the exact same constraint.
38. `IC::file()` holds **two different formats** depending on how the IC was loaded: `IC::load()` (stream deserialization of an already-saved circuit) strips the directory, storing a *bare filename*; `ICLoader::loadFile()` (the path used when a file-backed IC is freshly created/dragged in) stores the *absolute path*. `ICRegistry::findICsByFile()` matches via `QFileInfo(ic->file()) == QFileInfo(givenPath)`, which is exact-path equality, not same-basename-anywhere — a test that hand-constructs an IC with a *bare* `m_file` (via friend access) while the calling code being tested builds an *absolute* path to search for (as `ICController::embedICByFile()`/`makeSelfContained()` do) will silently find zero matches. Confirmed by tracing an actual test failure back through `IC.cpp`/`ICLoader.cpp`/`ICRegistry.cpp`, not assumed. When hand-building a file-backed IC fixture via friend access, match whichever load path the code under test actually mimics — `QDir(contextDir).absoluteFilePath(name)`, not a bare name, for anything downstream of `findICsByFile()`.
39. `QFile::moveToTrash()`'s failure mode is *not* reliably triggered by `chmod`'ing the containing directory read-only in this sandbox — confirmed empirically (still returns `true`) even with the directory, or the file itself, stripped of all permission bits (the current user remains the file's owner either way). What *does* reliably fail is a file this process has no ownership/permission relationship to at all: a real root-owned file in a root-only-writable directory (e.g. `/etc/hostname`). Deliberately picked a low-consequence file (not e.g. `/usr/bin/ls`) so that if this assumption is ever wrong in some other environment, the blast radius stays harmless — and verified the file untouched after the test ran, not just assumed safe.
40. A real `QMenu::exec()` popup **cannot** be driven to return from a headless/offscreen test via `action->trigger()` — that only emits the `QAction::triggered` signal, it does not drive `QMenuPrivate`'s internal close/return state machine, so `exec()`'s nested event loop blocks forever (confirmed via `gdb -p <pid> -batch -ex "thread apply all bt"` on the actually-hung process, per the project's standing "attach GDB, don't guess" rule — the backtrace showed the test thread parked inside `QMenuPrivate::exec()`). The fix is a **real synthetic mouse click** on the target action's geometry: `QTest::mouseClick(menu, Qt::LeftButton, Qt::NoModifier, menu->actionGeometry(action).center())` inside the `AutoDismisser` handler (found via `QApplication::topLevelWidgets()` once the menu becomes visible, same discovery mechanism as any other `AutoDismisser` use) — this goes through `QMenu`'s own real event handling, which both activates the action and makes `exec()` return it, exactly like a genuine user click. An older sibling file (`TestElementContextMenu.cpp`) had already independently concluded `ElementContextMenu::exec()` "can't be driven from a headless unit test" and tested its dispatch logic indirectly instead — that conclusion no longer needs to hold once this technique is used, though the two approaches aren't mutually exclusive.
41. `hasFocus()` on a widget parented to another widget (the common case — e.g. `ElementEditor(&workspace)`, a *child* of its `WorkSpace`) stays false even after a `setFocus()` call, unless the widget's actual **top-level window** (`w->window()`, which walks up the parent chain) is both `show()`n and made the active window — `QApplication::setActiveWindow()` only operates on top-level widgets, so calling it on the child itself is a silent no-op. Fix: `QWidget *top = w->window(); top->show(); w->show(); QApplication::setActiveWindow(top);` (mirrors `TestMainWindowGui.cpp`'s `createMW()` helper, generalized for a non-top-level target). Separately, a widget that starts life `setEnabled(false)`-and-`hide()`-in-constructor (as `ElementEditor` deliberately does, until a real selection enables it) will also silently swallow `setFocus()` while still disabled — Qt refuses focus to a disabled widget or its children — so a direct-call test of a focus-granting method (`renameAction()`/`changeTriggerAction()`) needs a real element selected first, not just the widget shown.
42. Under parallel `ctest` execution (each test class its own OS process, but real CPU contention across all of them), a **queued Qt focus-change event** isn't guaranteed to have been delivered by the time control returns synchronously from the call that triggered it (e.g. `contextMenu()` returning right after its internal `QMenu::exec()` unblocks) — confirmed by an isolated single-class run passing reliably 3/3 times while the same test failed once under the full 206-test parallel suite. Insert `QApplication::processEvents();` right after the triggering call and before asserting `hasFocus()`, rather than treating the occasional failure as unrelated flakiness (per `project_flaky_parallel_tests`) — the assertion is still checking real, non-fabricated state, just after giving the queued event a deterministic chance to land.
43. A synthetic `QTest::mouseClick()` on a real `QMenu` popup can silently fail to register on the intended action — found empirically, not assumed: a "Cut" action was located at its correct `actionGeometry()`, "clicked", yet neither the scene nor the undo stack changed, reproducing 3/3 times when run after another `exec()`+click in the same test but never in isolation (root-caused via layered `qWarning()` probes — topLevelWidgets list, undo-stack contents/text, the clicked element's `scene()` pointer — since this wasn't a hang GDB could backtrace). Fix: precede the click with a `QTest::mouseMove()` to the same point, priming `QMenuPrivate`'s own hover/current-action tracking; reliable in every ordering tried afterward. Same technique closes a related gap: entering a one-level-nested submenu (e.g. "Change color to..."/"Morph to...") needs a real hover (`QTest::mouseMove` on the parent action) rather than a manual `QMenu::popup()` call — `popup()` does make the submenu its own visible top-level widget (so a *second* poll can see it), but leaves the outer `exec()`'s internal active-popup chain unaware of it, so a click landing inside it never unblocks the outer call.
44. A real `QFileDialog`'s own accept-button auto-enable/disable bookkeeping (`QFileDialogPrivate::_q_updateOkButton()`) looks its button up by `QDialogButtonBox::Open`/`Save` *standard-button* identity — which, per pattern 40's sibling finding, this dialog's button never has under this Qt build. The practical consequence (confirmed via layered `qWarning()` probes on an actually-hung `exec()`, not assumed): once that bookkeeping leaves the button disabled, it can stay disabled *forever*, even after a valid filename is selected, hanging `exec()` non-deterministically (~1-in-3 runs, not reliably reproducible on demand) — this is qualitatively different from every other hang this sweep found, which were 100% reproducible until fixed. Since that auto-enable logic is Qt's own internal concern rather than anything the code under test controls, the robust fix is to force the button enabled (`button->setEnabled(true)`) immediately before the synthetic click rather than gating on `isEnabled()` — this still exercises the real dialog/`exec()`/`selectedFiles()` path being tested, just without depending on unrelated, racy Qt-internal bookkeeping. Separately, this also proves `QFileDialog::accept()` itself refuses to close when nothing is selected (clicking a force-enabled accept button with an empty filename just hangs `exec()` forever too) — don't attempt to test an "accepted with empty selection" branch by force-enabling past a disabled button; that specific combination is genuinely unreachable, not just hard.
45. A `return someTemplateCall([&]() -> ReturnType { ...multi-line lambda body... }, otherArgs);` call spanning many physical lines can show its *opening* line (the `someTemplateCall([&]() -> ReturnType {` part) as 0-hits while every statement inside the lambda body shows real hit counts — confirmed via a passing test that asserts on the lambda body's actual real-world effects (e.g. a real undo/redo state change), not just "the test passed". Same root cause class as patterns 2/8 (gcov misattributing a multi-line statement's "entry" credit), but for a lambda-taking call rather than chained `.arg()`/streamed logging — reformatting isn't practical here since the lambda body genuinely needs to stay multi-line. Exclude the opening line with a comment noting the body below it is the real, covered code; don't mistake this for the call never having run.
